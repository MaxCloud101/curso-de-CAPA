# Sync Phases

Las Sync Phases en Argo CD permiten controlar cuándo se ejecutan ciertos recursos durante un despliegue. Se implementan mediante hooks y son muy útiles para migraciones de base de datos, validaciones, smoke tests, notificaciones y tareas de limpieza.

## Flujo general

Durante un Sync, Argo CD sigue este orden:

```
PreSync
   ↓
Sync
   ↓
PostSync
```

Si algo falla en cualquier etapa:
```
SyncFail
```
puede ejecutarse automáticamente

## 1. PreSync

Se ejecuta antes de aplicar los manifiestos de la aplicación. Si cualquier hook PreSync falla, Argo detiene completamente el despliegue.

### Casos de uso

- Migraciones de base de datos
- Verificación de dependencias
- Validación de configuración
- Creación de recursos previos

Ejemplo:
```
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
  annotations:
    argocd.argoproj.io/hook: PreSync
spec:
  ...
```
Flujo
```
PreSync Job
    ↓
OK
    ↓
Deploy aplicación
```
o
```
PreSync Job
    ↓
FAIL
    ↓
Sync Failed
```

## 2. Sync

Es la fase principal.

Aquí Argo aplica:

- Deployments
- Services
- ConfigMaps
- Secrets
- Ingresses
- CRDs
- Cualquier manifiesto normal

Además, puedes definir hooks explícitos de tipo Sync.
```
metadata:
  annotations:
    argocd.argoproj.io/hook: Sync
```
Ejemplo

Un Job que coordine un despliegue complejo:
```
metadata:
  annotations:
    argocd.argoproj.io/hook: Sync
```

## 3. PostSync

Se ejecuta únicamente cuando:

1. El Sync terminó correctamente.
2. Los recursos están en estado Healthy.

Si un PostSync falla, la operación completa queda marcada como fallida.

### Casos de uso

- Smoke tests
- Tests de integración
- Notificaciones Slack
- Warmup de caché
- Validaciones finales

Ejemplo:

```
apiVersion: batch/v1
kind: Job
metadata:
  name: smoke-tests
  annotations:
    argocd.argoproj.io/hook: PostSync
```

Flujo:

```
Deploy OK
    ↓
Pods Healthy
    ↓
Smoke Tests
    ↓
Success
```

## 4. SyncFail

Se ejecuta cuando cualquier fase falla.

Puede fallar:
```
Un PreSync
Un Sync
Un PostSync
```
En cualquiera de esos casos se dispara el hook SyncFail.

Ejemplo:
```
metadata:
  annotations:
    argocd.argoproj.io/hook: SyncFail
```
Casos de uso
- Enviar alertas
- Limpiar recursos temporales
- Ejecutar rollback lógico
- Generar tickets automáticos

Flujo:
```
Deploy
   ↓
Error
   ↓
SyncFail Hook
   ↓
Notificación
```

## 5. Skip


No es realmente una fase de ejecución.

Le dice a Argo:

"No apliques este recurso."
```
metadata:
  annotations:
    argocd.argoproj.io/hook: Skip
```
Argo simplemente lo ignora

## 6. PostDelete (versiones recientes)

Se ejecuta después de eliminar completamente una Application. Está disponible en versiones modernas de Argo CD.

Ejemplo:
```
metadata:
  annotations:
    argocd.argoproj.io/hook: PostDelete
```
Útil para:

- Eliminar esquemas temporales
- Borrar recursos externos
- Limpieza final

Cómo se configura un Hook

Todo se basa en anotaciones:
```
metadata:
  annotations:
    argocd.argoproj.io/hook: PreSync
```
También puedes combinar varias:
```
metadata:
  annotations:
    argocd.argoproj.io/hook: PreSync,PostSync
```

## Sync Phases & Sync Waves

Mucha gente las confunde.

### Sync Phase

Define cuándo se ejecuta algo.
```
PreSync
Sync
PostSync
```

### Sync Wave

Define el orden dentro de una fase.
```
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "-1"
```
Ejemplo:
```
PreSync
 ├── Wave -1 → Crear DB
 ├── Wave 0  → Migración DB
 └── Wave 1  → Validación

Sync
 ├── Wave 0  → Backend
 └── Wave 1  → Frontend

PostSync
 └── Wave 0  → Smoke Tests
```
Argo ordena los recursos primero por Phase, luego por Wave, después por tipo de recurso y finalmente por nombre.
