# Remote Cluster Connection

## Arquitectura

Supongamos que ArgoCD está instalado en un cluster central:

```
                    +----------------+
                    |    Git Repo    |
                    +--------+-------+
                             |
                             v
+------------------------------------------------+
|          Cluster de Management                 |
|                                                 |
|  +---------------------------+                 |
|  |          ArgoCD           |                 |
|  +-------------+-------------+                 |
|                |                               |
+----------------+-------------------------------+
                 |
       +---------+---------+
       |                   |
       v                   v

+-------------+     +-------------+
| Cluster Dev |     | Cluster Prod|
+-------------+     +-------------+
```

ArgoCD mantiene una conexión API con cada cluster remoto para:

- Obtener el estado actual de los recursos.
- Comparar (Diff) contra Git.
- Aplicar cambios (Sync).
- Eliminar recursos (Prune).
- Obtener eventos y health checks.

## ¿Cómo registra ArgoCD un cluster?

Normalmente mediante:
```
argocd cluster add <contexto-kubeconfig>
```
Ejemplo:
```
argocd cluster add eks-prod
```
Lo que ocurre internamente es:

1. ArgoCD lee tu kubeconfig.
2. Se conecta al cluster remoto.
3. Crea un ServiceAccount.
4. Crea Roles/ClusterRoles.
5. Crea un Secret en el namespace argocd.

Ese Secret contiene:
```
kind: Secret
metadata:
  labels:
    argocd.argoproj.io/secret-type: cluster
```

## ¿Dónde guarda la información?

En el cluster donde está instalado ArgoCD.

Por ejemplo:
```
kubectl get secrets -n argocd
```
Verás algo similar a:
```
cluster-eks-dev
cluster-eks-prod
cluster-aks-prod
```
Cada Secret contiene:
```
data:
  name:
  server:
  config:
```
Donde:

- name = nombre lógico
- server = URL del API Server
- config = credenciales

## ¿Cómo se autentica?

Depende del tipo de cluster.

### 1. Service Account Token (Tradicional)

ArgoCD crea:
```
ServiceAccount
ClusterRole
ClusterRoleBinding
```
Y utiliza el token asociado.
```
ArgoCD
   |
Bearer Token
   |
   v
Remote Cluster API
```
Es el mecanismo más simple.

### 2. EKS + IAM Roles

Muy común en AWS.

En lugar de guardar tokens:
```
ArgoCD
   |
AssumeRole
   |
AWS IAM
   |
   v
EKS
```
El Secret del cluster contiene:
```
awsAuthConfig:
  clusterName: prod
  roleARN: arn:aws:iam::123456789:role/argocd
```
Cuando necesita acceder:

1. ArgoCD asume el rol.
2. Genera credenciales temporales.
3. Obtiene token EKS.
4. Llama al API Server.

Más seguro que tokens permanentes.

### 3. AKS + Azure Workload Identity

Patrón equivalente para Azure.
```
ArgoCD
   |
Azure Federated Identity
   |
   v
AKS
```
### 4. GKE + Workload Identity

Patrón equivalente para Google Cloud.
```
ArgoCD
   |
GCP Identity
   |
   v
GKE
```

## ¿Qué permisos necesita?

Mínimamente:
```
get
list
watch
create
update
patch
delete
```
Sobre los recursos que va a gestionar.

Muchas organizaciones evitan:
```
cluster-admin
```
Y crean permisos específicos.

Ejemplo:
```
ArgoCD
  |
  +-- namespace team-a
  |
  +-- namespace team-b
```
Solo administra esos namespaces.
