# Remote Cluster Connection

## Arquitectura

Supongamos que ArgoCD está instalado en un cluster central:

```
                    +----------------+
                    |    Git Repo    |
                    +--------+-------+
                             |
                             v
+------------------------------------------------+
|          Cluster de Management                 |
|                                                 |
|  +---------------------------+                 |
|  |          ArgoCD           |                 |
|  +-------------+-------------+                 |
|                |                               |
+----------------+-------------------------------+
                 |
       +---------+---------+
       |                   |
       v                   v

+-------------+     +-------------+
| Cluster Dev |     | Cluster Prod|
+-------------+     +-------------+
```

ArgoCD mantiene una conexión API con cada cluster remoto para:

- Obtener el estado actual de los recursos.
- Comparar (Diff) contra Git.
- Aplicar cambios (Sync).
- Eliminar recursos (Prune).
- Obtener eventos y health checks.

## ¿Cómo registra ArgoCD un cluster?

Normalmente mediante:
```
argocd cluster add <contexto-kubeconfig>
```
Ejemplo:
```
argocd cluster add eks-prod
```
Lo que ocurre internamente es:

1. ArgoCD lee tu kubeconfig.
2. Se conecta al cluster remoto.
3. Crea un ServiceAccount.
4. Crea Roles/ClusterRoles.
5. Crea un Secret en el namespace argocd.

Ese Secret contiene:
```
kind: Secret
metadata:
  labels:
    argocd.argoproj.io/secret-type: cluster
```

## ¿Dónde guarda la información?

En el cluster donde está instalado ArgoCD.

Por ejemplo:
```
kubectl get secrets -n argocd
```
Verás algo similar a:
```
cluster-eks-dev
cluster-eks-prod
cluster-aks-prod
```
Cada Secret contiene:
```
data:
  name:
  server:
  config:
```
Donde:

- name = nombre lógico
- server = URL del API Server
- config = credenciales
