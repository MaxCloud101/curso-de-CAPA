# Projects

Project (AppProject) sirve para agrupar aplicaciones y definir qué pueden hacer. Es una capa de gobernanza que permite controlar:

- Qué repositorios Git pueden usar las aplicaciones.
- A qué clústeres pueden desplegar.
- En qué namespaces pueden desplegar.
- Qué recursos Kubernetes están permitidos o prohibidos.
- Qué equipos (roles) tienen acceso.

Componentes de un AppProject

## 1 Metadata

```
metadata:
  name: platform-team
  namespace: argocd
```

- name: nombre del proyecto.
- namespace: normalmente siempre es argocd, porque ahí corre Argo CD.

El proyecto se referenciará desde las aplicaciones:

```
spec:
  project: platform-team
```

## 2 Description

```
description: Proyecto para aplicaciones del equipo de plataforma
```

Descripción informativa que aparece en la UI de Argo CD.


## 3 Source Repositories

```
sourceRepos:
  - 'https://github.com/company/platform-apps.git'
  - 'https://github.com/company/helm-charts.git'
```

Define qué repositorios Git están permitidos.

Si una aplicación intenta usar otro repositorio:
```
repoURL: https://github.com/company/otro-repo.git
```
Argo CD la rechazará.

También puedes permitir todos:
```
sourceRepos:
  - '*'
```
Aunque en producción no suele ser recomendable.

## 4 Destinations

```
destinations:
  - namespace: production
    server: https://kubernetes.default.svc

  - namespace: staging
    server: https://kubernetes.default.svc
```

Controla dónde pueden desplegar las aplicaciones.

- namespace: Namespace permitido.
- server: Identifica el clúster destino.

Si tienes múltiples clústeres:

```
destinations:
  - namespace: production
    server: https://prod-cluster.company.com

  - namespace: staging
    server: https://staging-cluster.company.com
```

## 5 Cluster Resource Whitelist

```
clusterResourceWhitelist:
  - group: '*'
    kind: '*'
```

Controla recursos Cluster Scope.

Ejemplos:

- ClusterRole
- ClusterRoleBinding
- Namespace
- CustomResourceDefinition (CRD)
- StorageClass

Con:

```
group: '*'
kind: '*'
```

se permite todo.

### Restringir CRDs únicamente

clusterResourceWhitelist:
  - group: apiextensions.k8s.io
    kind: CustomResourceDefinition

## 7 Namespace Resource Whitelist

```
namespaceResourceWhitelist:
  - group: '*'
    kind: '*'
```

Controla recursos dentro de namespaces.

Ejemplos:

- Deployment
- Service
- ConfigMap
- Secret
- Ingress

### Permitir únicamente ciertos recursos

```
  - group: apps
    kind: Deployment

  - group: ""
    kind: Service

  - group: ""
    kind: ConfigMap
```

### 8 Resource Blacklist

También puedes prohibir recursos.

```
namespaceResourceBlacklist:
  - group: ""
    kind: Secret
```

Así ninguna aplicación del proyecto podrá crear Secrets.

### 9 Orphaned Resources
```
orphanedResources:
  warn: true
```

Hace que Argo CD avise cuando detecta recursos que existen en Kubernetes pero ya no están definidos en Git.

Ejemplo:

```
deployment-a
service-a
configmap-viejo
```

Si eliminas configmap-viejo del repositorio pero sigue existiendo en el clúster, Argo CD lo marcará como huérfano (orphaned).


### 10 Roles
Los roles permiten RBAC específico por proyecto.

```
roles:
  - name: developers
```
Crea el rol:

developers

## 11 Policies

```
policies:
  - p, proj:platform-team:developers, applications, get, platform-team/*, allow
```

Permite ver aplicaciones.

```
- p, proj:platform-team:developers, applications, sync, platform-team/*, allow
```

Permite ejecutar Sync.

### Formato:

```
p, <subject>, <resource>, <action>, <object>, allow
```
