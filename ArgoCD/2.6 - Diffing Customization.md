# Diffing Customization

El Diffing Customization en Argo CD sirve para indicarle a Argo:

"Ignora estas diferencias porque sé que Kubernetes o algún controlador las va a modificar automáticamente."

Sin esta funcionalidad, muchas aplicaciones aparecen constantemente como OutOfSync aunque realmente no haya ningún problema.

## ¿Por qué ocurre?

Argo CD compara:
```
Git (Desired State)
        VS
Cluster (Live State)
```
Si encuentra diferencias:
```
Git != Live
```
marca la aplicación como:
```
OutOfSync
```
El problema es que Kubernetes y muchos operadores modifican recursos después de aplicarlos.

Por ejemplo:
```
spec:
  replicas: 3
```
Podría terminar en el cluster como:
```
spec:
  replicas: 5
```
porque un HPA lo cambió.

Argo verá:
```
3 != 5
```
y marcará OutOfSync.

## ignoreDifferences

La personalización más utilizada es:
```
spec:
  ignoreDifferences:
```
Permite ignorar campos específicos durante el diff.

### Ejemplo 1: Ignorar replicas

Si usas HPA:
```
apiVersion: apps/v1
kind: Deployment
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
```
Ahora Argo ignorará:
```
spec:
  replicas
```
durante la comparación.

### Ejemplo 2: Ignorar imágenes

Imagina que una herramienta externa actualiza las imágenes.

Git:
```
image: nginx:1.0
```
Cluster:
```
image: nginx:1.1
```
Puedes ignorarlo:
```
ignoreDifferences:
  - group: apps
    kind: Deployment
    jqPathExpressions:
      - .spec.template.spec.containers[].image
```

#### jsonPointers

Utiliza la especificación JSON Pointer.

Ejemplos:

```
/spec/replicas
/metadata/annotations
/spec/template/spec/containers
```

#### jqPathExpressions

Más potente que JSON Pointer.

Permite filtrar estructuras complejas.

Ejemplo:
```
jqPathExpressions:
  - .spec.template.spec.initContainers[]
```
o
```
jqPathExpressions:
  - .webhooks[]?.clientConfig.caBundle
```
Muy usado para certificados generados automáticamente.

#### managedFieldsManagers

Desde Kubernetes 1.18 existe:
```
metadata:
  managedFields:
```
Ejemplo:
```
managedFields:
  - manager: kube-controller-manager
```
Puedes decir:
```
ignoreDifferences:
  - group: '*'
    kind: '*'
    managedFieldsManagers:
      - kube-controller-manager
```
Entonces Argo ignorará cualquier campo gestionado por:
```
kube-controller-manager
```

## Aplicación vs Sistema

Puedes configurar el diff customization a dos niveles.

### Por aplicación
```
spec:
  ignoreDifferences:
```
Solo afecta esa Application.

### Globalmente

En el ConfigMap de Argo:
```
argocd-cm
```

```
data:
  resource.customizations: |
```

Ejemplo:

```
resource.customizations: |
  apps/Deployment:
    ignoreDifferences: |
      jsonPointers:
      - /spec/replicas
```

Afectará a todos los Deployments gestionados por Argo CD.

## RespectIgnoreDifferences

Hay un detalle importante.

Por defecto:
```
ignoreDifferences
```
solo afecta el diff.

Durante un Sync, Argo sigue aplicando el manifiesto completo.

Para que también lo respete en el Sync:
```
syncPolicy:
  syncOptions:
    - RespectIgnoreDifferences=true
```
Ejemplo:
```
spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas

  syncPolicy:
    syncOptions:
      - RespectIgnoreDifferences=true
```
Ahora Argo:

Ignora el campo en el diff.
No intenta restaurarlo durante el sync.

## Cuándo usar Diff Customization

Úsalo cuando el recurso sea modificado por:

- HPA
- VPA
- Cert Manager
- Istio
- Linkerd
- External Secrets Operator
- AWS Load Balancer Controller
- Controladores personalizados
- Mutating Admission Webhooks

Evítalo para ocultar errores reales de configuración. Si ignoras demasiados campos, Argo puede mostrar una aplicación como Synced cuando en realidad existe una desviación importante respecto a Git.

Una buena práctica es ignorar únicamente los campos que sabes con certeza que son gestionados por otro controlador y documentar el motivo de cada regla de ignoreDifferences.
