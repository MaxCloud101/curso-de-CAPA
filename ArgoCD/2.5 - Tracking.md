# Tracking

Las tracking strategies definen cómo el campo targetRevision sigue los cambios en Git o en un repositorio Helm. Básicamente responden a la pregunta:

"¿Qué versión del código o manifiestos debe desplegar Argo CD?"

# Git Tracking

## 1. Branch Tracking (HEAD / Rama)

Es la estrategia más común en entornos de desarrollo.
```
spec:
  source:
    repoURL: https://github.com/company/app.git
    targetRevision: main
```
o
```
targetRevision: HEAD
```

### ¿Cómo funciona?

Argo CD monitorea continuamente la punta de la rama (main, develop, etc.).

Si alguien hace:

```
git push origin main
```

Argo detecta el nuevo commit y marca la aplicación como OutOfSync. Si tienes auto-sync, desplegará automáticamente.


## 2. Tag Tracking

En lugar de seguir una rama, sigues un tag.
```
targetRevision: v1.2.0
```
### ¿Cómo funciona?

Argo CD usa los manifiestos asociados a ese tag Git.

Por ejemplo:
```
git tag v1.2.0
git push origin v1.2.0
```
Argo desplegará exactamente ese estado.

Caso típico
### Producción
```
targetRevision: v2.3.1
```
Solo se despliega una nueva versión cuando cambias el tag o actualizas el targetRevision.

## 3. Commit Pinning

Es la estrategia más estricta.
```
targetRevision: a4d5c9e72f8c3d1
```
### ¿Cómo funciona?

La aplicación queda fijada a un commit específico.

Aunque lleguen nuevos commits a la rama:
```
main
 ├── commit A
 ├── commit B
 └── commit C
```
Si Argo está apuntando a:

targetRevision: commitA

nunca se moverá automáticamente.

# Helm Tracking

Cuando el origen es un repositorio Helm, targetRevision representa la versión del chart.

## Versión fija
```
targetRevision: 1.2.3
```
Solo se usa esa versión.

## Seguir patches
```
targetRevision: 1.2.*
```
Si aparecen:
```
1.2.1
1.2.2
1.2.3
```
Argo tomará automáticamente la más reciente dentro de la serie 1.2.x.

## Seguir minor versions
```
targetRevision: 1.*
```
Permitirá:
```
1.0.0
1.1.0
1.5.2
1.9.8
```
pero nunca:
```
2.0.0
```

Estrategia recomendada por ambiente

| Ambiente   | Estrategia                   |
| ---------- | ---------------------------- |
| Dev        | `HEAD` o `develop`           |
| QA         | Tag semántico (`v1`, `v1.2`) |
| Staging    | Tag específico (`v1.2.3`)    |
| Producción | Commit SHA o Tag inmutable   |

## Un detalle importante

Evita tener una rama y un tag con el mismo nombre:

```
release-1.0   (branch)
release-1.0   (tag)
```

Argo CD puede resolver la referencia de forma ambigua y provocar reconciliaciones inesperadas. La documentación recomienda usar referencias completas como:
```
targetRevision: refs/heads/release-1.0
```
o
```
targetRevision: refs/tags/release-1.0
```
para evitar problemas.
