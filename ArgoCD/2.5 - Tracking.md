# Tracking

Las tracking strategies definen cómo el campo targetRevision sigue los cambios en Git o en un repositorio Helm. Básicamente responden a la pregunta:

"¿Qué versión del código o manifiestos debe desplegar Argo CD?"

## 1. Branch Tracking (HEAD / Rama)

Es la estrategia más común en entornos de desarrollo.
```
spec:
  source:
    repoURL: https://github.com/company/app.git
    targetRevision: main
```
o
```
targetRevision: HEAD
```

### ¿Cómo funciona?

Argo CD monitorea continuamente la punta de la rama (main, develop, etc.).

Si alguien hace:

```
git push origin main
```

Argo detecta el nuevo commit y marca la aplicación como OutOfSync. Si tienes auto-sync, desplegará automáticamente.


## 2. Tag Tracking

En lugar de seguir una rama, sigues un tag.
```
targetRevision: v1.2.0
```
### ¿Cómo funciona?

Argo CD usa los manifiestos asociados a ese tag Git.

Por ejemplo:
```
git tag v1.2.0
git push origin v1.2.0
```
Argo desplegará exactamente ese estado.

Caso típico
### Producción
```
targetRevision: v2.3.1
```
Solo se despliega una nueva versión cuando cambias el tag o actualizas el targetRevision.
