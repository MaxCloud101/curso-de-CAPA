# Sync

En ArgoCD, un sync (o sincronización) es el proceso automático o manual que iguala el estado de tu clúster de Kubernetes con los archivos de configuración (manifiestos) definidos en el repositorio de Git.

| Característica         | Sync Manual                            | Sync Automático                                 |
| ---------------------- | -------------------------------------- | ----------------------------------------------- |
| Quién ejecuta el Sync  | Un usuario o pipeline                  | ArgoCD                                          |
| Cambio en Git          | Marca la app como `OutOfSync`          | Sincroniza automáticamente                      |
| Drift en el cluster    | Solo se corrige si alguien hace Sync   | Puede corregirse automáticamente con `selfHeal` |
| Control de despliegues | Máximo control                         | Máxima automatización                           |
| Uso típico             | Producción regulada o con aprobaciones | GitOps puro, Dev, Staging                       |

## Sync Manual
```
Git --> ArgoCD detecta cambio --> OutOfSync --> Espera Sync manual
```

## Sync Automatico

```
Git --> ArgoCD detecta cambio --> OutOfSync --> Sync automático --> Synced
```

Cuando alguien hace un commit en Git:

1. ArgoCD detecta el nuevo commit.
2. Compara los manifiestos generados con los recursos del cluster.
3. Si encuentra diferencias (OutOfSync), ejecuta automáticamente un Sync.
4. Aplica los cambios al cluster.

### Configuración básica
```
spec:
  syncPolicy:
    automated: {}
```
o explícitamente:
```
spec:
  syncPolicy:
    automated:
      enabled: true
```

### Componentes del Auto Sync

#### 1. Auto Sync simple

```
spec:
  syncPolicy:
    automated: {}
```

Comportamiento:

- Aplica cambios nuevos que aparecen en Git.
- No elimina recursos.
- No corrige cambios manuales hechos en Kubernetes.

#### 2. Prune
```
spec:
  syncPolicy:
    automated:
      prune: true
```

Si eliminas algo del repositorio Git:
```
Git:
deployment.yaml eliminado
```
ArgoCD también lo eliminará del cluster automáticamente.

Ejemplo:
```
Git:
- deployment
- service

Commit siguiente:
- deployment

Resultado:
service eliminado del cluster
```

#### 3. Self Heal
```
spec:
  syncPolicy:
    automated:
      selfHeal: true
```
Esta opción es una de las más importantes.

Supongamos que Git tiene:
```
replicas: 3
```
Pero alguien ejecuta:
```
kubectl scale deployment nginx --replicas=10
```

El cluster queda distinto a Git.

```
Con selfHeal: true:
```

```
Git = 3 replicas
Cluster = 10 replicas

↓
ArgoCD detecta drift

↓
ArgoCD vuelve a aplicar

↓
Cluster = 3 replicas
```
Es decir, Git vuelve a ser la fuente de verdad.

#### 4. Allow Empty

```
spec:
  syncPolicy:
    automated:
      prune: true
      allowEmpty: true
```

Normalmente ArgoCD protege contra errores.

Si por accidente Git genera:
```
0 recursos
```

ArgoCD no borra todo el entorno.

Con allowEmpty: true le dices:
```
"Sí, si Git está vacío, elimina todo."
```

### Ejemplo recomendado para GitOps

```
spec:
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

Este es probablemente el modo más usado en producción.

Beneficios:

- Despliegues automáticos.
- Limpieza automática de recursos eliminados.
- Corrección automática de drift.
- Git es la única fuente de verdad.

### Cosas importantes que mucha gente no sabe

#### Auto Sync solo corre cuando la App está OutOfSync

Si la aplicación está:
```
Synced
```
No hará nada.

Cuando detecta:

```
OutOfSync
```

Dispara el Sync automático.

#### No reintenta infinitamente un Sync fallido

Si un Sync falla para un commit específico:

```
Commit abc123
↓
Sync falla
```

ArgoCD no seguirá intentando indefinidamente sobre ese mismo commit. Necesitas un nuevo commit o configurar políticas de retry.

#### No puedes hacer Rollback mientras Auto Sync está activo

Imagina:
```
Rollback a versión anterior
```

ArgoCD inmediatamente verá que Git apunta a otra versión y volverá a sincronizar.

Por eso, para hacer rollback normalmente:

1. Deshabilitas Auto Sync.
2. Haces rollback.
3. O mejor aún, haces revert del commit en Git.


### Componentes del Sync Options

Las Sync Options son parámetros que modifican el comportamiento del proceso de sincronización de ArgoCD. Se configuran dentro de:

```
spec:
  syncPolicy:
    syncOptions:
      - Opcion=valor
```

o como anotación:

```
metadata:
  annotations:
    argocd.argoproj.io/sync-options: Opcion=valor
```

### syncOptions más utilizadas

#### 1. CreateNamespace

```
syncOptions:
  - CreateNamespace=true
```
Si el namespace destino no existe, ArgoCD lo crea automáticamente.

Sin esta opción:

```
namespace: mi-app

Sync Failed:
namespaces "mi-app" not found
```

Con esta opción:

```
ArgoCD crea el namespace
↓
Despliega la aplicación
```

Muy común en GitOps.

#### 2. PruneLast

```
syncOptions:
  - PruneLast=true
```

Hace que los recursos a eliminar se borren al final del Sync.

Sin PruneLast:

```
1. Borra recursos
2. Crea recursos nuevos
```

Con PruneLast:

```
1. Crea/actualiza recursos
2. Espera que estén Healthy
3. Borra recursos obsoletos
```

Ejemplo típico:

```
Service antiguo
↓
Deployment nuevo
```

Evita downtime durante migraciones.


#### 3. ApplyOutOfSyncOnly

```
syncOptions:
  - ApplyOutOfSyncOnly=true
```

Solo aplica recursos que estén realmente fuera de sincronización.

Sin esta opción:

```
100 recursos
↓
kubectl apply sobre los 100
```

Con esta opción:

```
100 recursos
↓
Solo 3 cambiaron
↓
Apply únicamente a esos 3
```

Reduce carga sobre el API Server.

Muy útil en aplicaciones grandes.

#### 4. RespectIgnoreDifferences

```
syncOptions:
  - RespectIgnoreDifferences=true
```

Hace que ArgoCD respete las reglas de ignoreDifferences también durante el Sync.

Ejemplo:

```
ignoreDifferences:
- group: apps
  kind: Deployment
  jsonPointers:
  - /spec/replicas
```

Si alguien cambia replicas manualmente:

```
Git: 3
Cluster: 5
```

ArgoCD ignorará esa diferencia.

Sin esta opción puede seguir intentando corregir algunos campos durante el Sync.

#### 5. Replace

```
syncOptions:
  - Replace=true
```

En vez de
```
kubectl apply
```

utiliza:

```
kubectl replace
```

o

```
kubectl create
```

Se usa cuando:

- Objetos muy grandes.
- Problemas con anotaciones.
- Recursos que exceden el límite de tamaño de kubectl apply.

No suele ser necesario en la mayoría de casos.

#### 6. Force

```
syncOptions:
  - Force=true
```

Fuerza recreación de recursos.

Equivalente conceptual:

```
kubectl delete
kubectl create
```

Útil para:

- Jobs.
- Recursos que deben recrearse cada vez.

Ejemplo:

```
kind: Job
```

Cada Sync ejecuta nuevamente el Job.

Debe usarse con cuidado porque puede generar interrupciones.
