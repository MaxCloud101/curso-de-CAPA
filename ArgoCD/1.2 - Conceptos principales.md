# Conceptos principales

## Application

Application es el recurso que le dice a ArgoCD:

> "Este repositorio Git, en esta ruta, debe desplegarse en este clúster y namespace."

Es el objeto que conecta:
```
Git Repository → Kubernetes Cluster
```
Sin una Application, ArgoCD no sabe qué desplegar, desde dónde ni a dónde.

Ejemplo mental:

```
ArgoCD:
"Toma los YAML del repositorio X,
usa la carpeta Y,
y sincronízalos en namespace Z."
```

Ejemplo real:

```
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nginx-app
  namespace: argocd

spec:
  project: default

  source:
    repoURL: https://github.com/company/k8s-apps
    targetRevision: main
    path: nginx

  destination:
    server: https://kubernetes.default.svc
    namespace: production

  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

## Project

Project (AppProject) es un recurso que sirve para agrupar aplicaciones y definir reglas o límites sobre lo que esas aplicaciones pueden hacer.

Piensa en él como una política de gobernanza o seguridad para Applications.

Sin proyectos personalizados, todas las aplicaciones usan:

```
project: default
```

**Analogía sencilla**

Imagina una empresa:

```
Empresa
│
├── Equipo Desarrollo
│      └── Puede desplegar solo en namespace dev
│
├── Equipo Producción
│      └── Puede desplegar solo en prod
│
└── Equipo Seguridad
       └── Puede usar repositorios específicos
```

Cada equipo sería un Project.

Las Applications pertenecen a un Project:

```
Project
   └── Application
          └── Deployment Kubernetes
```

**¿Para qué sirve un Project?**

Permite controlar:

- Qué repositorios Git pueden usarse
- Qué clusters pueden usarse
- Qué namespaces pueden usarse
- Qué recursos Kubernetes están permitidos
- Roles RBAC
- Permisos por equipo

Ejemplo de Project

```
apiVersion: argoproj.io/v1alpha1
kind: AppProject

metadata:
  name: development
  namespace: argocd

spec:

  sourceRepos:
  - https://github.com/company/*

  destinations:
  - namespace: dev
    server: https://kubernetes.default.svc

  clusterResourceWhitelist:
  - group: '*'
    kind: '*'
```

## Estado deseado vs estado actual

Estado deseado: descrito en git.
Estado actual: lo que realmente está funcionando.

La reconciliación (reconciliation) es el proceso mediante el cual una herramienta como Argo CD compara continuamente:
```
Desired State (estado deseado)
VS
Actual State / Live State (estado real)
```
y toma acciones para hacer que ambos coincidan.

## Sync

Es el proceso de aplicar los cambios necesarios para que el estado actual y el estado deseado sean los mismos.

## Refresh

Refresh significa forzar a ArgoCD a volver a consultar el estado actual de una Application, comparando nuevamente:

```
Git (estado deseado)
        VS
Kubernetes (estado real)
```

El refresh no despliega cambios automáticamente (eso es Sync). Solo actualiza la información que ArgoCD conoce.
